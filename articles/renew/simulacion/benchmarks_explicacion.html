1. Visión general del archivo

benchmarks.py implementa los dos algoritmos de referencia (benchmarks) descritos en la Sección 4.4 del paper. Su propósito es tener una base de comparación contra los algoritmos DDPG/MADDPG, para demostrar las ventajas del aprendizaje por refuerzo profundo.

🧮 RBMAgent

Rule-Based Model. No aprende nada. Aplica reglas deterministas de prioridad: carga el ESS cuando hay exceso de energía renovable y descarga cuando hay déficit. Orden fijo: SC → LIB → VRB.

🧠 MADQNSystem

Multi-Agent Deep Q-Network. Tres agentes DQN independientes (uno por ESS). Cada uno aprende su propia función Q con espacio de acciones discreto (5 niveles) usando Double DQN + replay buffer.

Estructura de clases

2. RBMAgent — Agente basado en reglas

El Rule-Based Model es la línea base más simple posible: no tiene red neuronal, no aprende, y siempre toma la decisión "físicamente obvia": si sobra energía renovable, carga; si falta, descarga.

Orden de prioridad (ESS_ORDER)

El paper especifica que el SuperCaps (SC) responde más rápido, seguido del Litio (LIB) y el Vanadio Redox (VRB). Esto se codifica como índices de la lista de ESS del entorno:

Prioridad 1
SC (idx=2)
Prioridad 2
LIB (idx=0)
Prioridad 3
VRB (idx=1)
ESS_ORDER = [2, 0, 1]   # SC=idx2, LIB=idx0, VRB=idx1

Lógica de select_actions()

Recibe como parámetros el estado físico actual y devuelve una acción por cada ESS en el rango [-1, 1]:

demand, pv, wt, ess_charges
→
surplus = pv+wt−demand
→
surplus > 0 ?
→
Cargar SC→LIB→VRB
/
Descargar SC→LIB→VRB
def select_actions(self, demand, pv_mw, wt_mw, ess_charges, ...):
    surplus = pv_mw + wt_mw - demand  # positivo = exceso renovable
    actions = [0.0, 0.0, 0.0]

    if surplus > 0:                     # cargar en orden de prioridad
        remaining = surplus
        for idx in self.ESS_ORDER:
            room = ess_c_max - ess_charges[idx]     # espacio disponible
            charge_mw = min(remaining, ess_x_max, room)
            actions[idx] = charge_mw / ess_x_max   # normalizar a [-1,1]
            remaining -= charge_mw
    else:                              # descargar en orden de prioridad
        deficit = -surplus
        for idx in self.ESS_ORDER:
            discharge_mw = min(deficit, ess_x_max, ess_charges[idx])
            actions[idx] = -(discharge_mw / ess_x_max)
            deficit -= discharge_mw

    return actions
⚠️
Limitación clave: el RBM solo maximiza el uso de renovables pero ignora completamente los precios de mercado. No sabe si la electricidad vale £16/MWh o £144/MWh en ese momento. Por eso los algoritmos RL lo superan ampliamente en ahorros económicos.

3. _DQNNetwork — Red neuronal Q

Subclase privada de nn.Module. Implementa la arquitectura de red neuronal que estima los valores Q para cada acción discreta. El paper especifica 2 capas ocultas de 256 neuronas con activación ReLU (Table 2).

class _DQNNetwork(nn.Module):
    def __init__(self, obs_dim, n_actions, hidden_dim=256):
        self.net = nn.Sequential(
            nn.Linear(obs_dim, hidden_dim), nn.ReLU(),   # capa 1
            nn.Linear(hidden_dim, hidden_dim), nn.ReLU(),# capa 2
            nn.Linear(hidden_dim, n_actions),            # salida: Q(s,a_i)
        )

    def forward(self, x):
        return self.net(x)  # devuelve tensor [batch, n_actions]
💡
La red devuelve un valor Q para cada acción posible en el paso actual. La acción elegida es argmax(Q). Con 5 acciones la dimensión de salida es [batch, 5].

4. _DQNSingleAgent — Agente DQN independiente

Implementa un agente Double DQN completo para un único ESS. Cada agente es completamente independiente: tiene su propia red, su propio target network, su propio optimizer y su propio replay buffer. Sigue el enfoque de Tampuu et al.

Componentes internos

🔵 online (red activa)

Se actualiza en cada paso de entrenamiento. Se usa para seleccionar acciones y calcular los Q-values actuales.

🟣 target (red objetivo)

Copia de la red online actualizada suavemente con τ=0.005. Proporciona los Q-targets estables en el cálculo de la pérdida Double DQN.

🟢 buffer (deque)

Replay buffer circular de 100,000 transiciones (s, a, r, s', done). Rompe las correlaciones temporales muestreando mini-batches aleatorios.

Exploración ε-greedy con decaimiento exponencial

El epsilon decae exponencialmente desde 1.0 (exploración total) hasta 0.05 (casi siempre greedy): 

# ε = ε_end + (ε_start - ε_end) × e^(−step / eps_decay)
@property
def epsilon(self) -> float:
    return self.eps_end + (self.eps_start - self.eps_end) * \
           np.exp(-self._step / self.eps_decay)

# Acción:
#   - con prob ε  → acción aleatoria (exploración)
#   - con prob 1-ε → argmax Q(s, ·)   (explotación)
ℹ️
Con eps_decay=5000, el agente alcanza ε≈0.37 a los 5,000 pasos, y ε≈0.05 (prácticamente greedy) alrededor de los 20,000 pasos.

Double DQN — Cálculo del target Q

La diferencia clave del Double DQN respecto al DQN clásico: la red online elige la acción óptima en el siguiente estado, pero el valor de esa acción lo estima la red target. Esto evita la sobreestimación sistemática de Q: 

with torch.no_grad():
    # 1) Red online elige la mejor acción en s' (evita maximization bias)
    next_actions = self.online(next_obs_t).argmax(dim=1, keepdim=True)

    # 2) Red target evalúa el valor de esa acción
    next_q = self.target(next_obs_t).gather(1, next_actions)

    # 3) Ecuación de Bellman:
    target_q = rew_t + (1.0 - done_t) * self.gamma * next_q

# 4) Pérdida MSE entre Q actual y Q target
current_q = self.online(obs_t).gather(1, act_t)
loss = F.mse_loss(current_q, target_q)

# 5) Actualización
self.opt.zero_grad()
loss.backward()
nn.utils.clip_grad_norm_(self.online.parameters(), 1.0)  # gradient clipping
self.opt.step()

# 6) Soft update del target network: θ_target ← τ·θ_online + (1-τ)·θ_target
for p, tp in zip(self.online.parameters(), self.target.parameters()):
    tp.data.copy_(self.tau * p.data + (1 - self.tau) * tp.data)

5. _discrete_action_to_continuous — Mapeo de acciones

El DQN trabaja con acciones discretas (índices 0..4), pero el entorno HESSMicrogridEnv espera acciones continuas en [−1, 1]. Esta función hace el mapeo lineal: 

def _discrete_action_to_continuous(idx: int, n_actions: int = 5) -> float:
    return -1.0 + 2.0 * idx / (n_actions - 1)

Tabla de mapeo (n_actions=5)

Índice discretoAcción continuaSignificado físico
0−1.0Descarga máxima (−X_max MW)
1−0.5Descarga media
20.0Inactivo (ni carga ni descarga)
3+0.5Carga media
4+1.0Carga máxima (+X_max MW)
💡
El paper menciona también una variante Rainbow con 9 acciones discretas, cubiertas por el mismo código cambiando n_actions=9. La fórmula genera automáticamente pasos de 0.25 en vez de 0.5.

6. MADQNSystem — Sistema multi-agente DQN

MADQNSystem es el orquestador: instancia 3 agentes DQN independientes (uno por ESS) y expone la misma interfaz que los sistemas MADDPG/DDPG para que case_study_1.py los llame de forma transparente.

Agente 0
LIB (Litio-Ion)
Agente 1
VRB (Vanadio Redox)
Agente 2
SC (SuperCaps)

Método select_actions()

Recibe una lista de observaciones (una por agente) y devuelve acciones continuas para el entorno:

def select_actions(self, obs_list: List[np.ndarray]) -> List[float]:
    actions = []
    for i, agent in enumerate(self.agents):
        idx = agent.select_action(obs_list[i])           # ε-greedy → índice
        actions.append(_discrete_action_to_continuous(idx, self.n_actions))
    return actions

Método push()

Almacena la transición. Dado que el entorno opera con acciones continuas, se recalcula el índice discreto más cercano antes de guardar en el buffer:

def push(self, obs_list, actions_cont, rewards, next_obs_list, done):
    for i, agent in enumerate(self.agents):
        cont = actions_cont[i]
        # invertir la fórmula de mapeo: idx = round((cont+1)/2 × (n-1))
        idx = int(round((cont + 1.0) / 2.0 * (self.n_actions - 1)))
        idx = max(0, min(self.n_actions - 1, idx))
        agent.push(obs_list[i], idx, rewards[i], next_obs_list[i], done)
ℹ️
MADQN es "independiente": cada agente toma decisiones sin conocer las acciones de los otros. Esto es lo que diferencia a MADQN de MADDPG, donde cada crítico sí tiene acceso a las acciones de todos los agentes (centralised training, decentralised execution).

7. Comparativa RBM vs MADQN vs MADDPG

Los resultados del paper (Tabla 3) muestran cómo los algoritmos RL superan al benchmark basado en reglas:

Algoritmo Tipo Acciones Ahorros anuales (CS1) Fortaleza
RBM Reglas fijas Continuas (calculadas) Bajo (no optimiza precio) Determinista, sin training
MADQN RL discreto 5 niveles discretos Medio Aprende dinámica de precios
MADDPG RL continuo multi-agente Continuas [−1,1] Alto Coordinación centralizada en entrenamiento
💡
El RBM sirve como "piso": si un algoritmo RL no supera al RBM, su entrenamiento ha fallado. En nuestras pruebas, incluso con solo 5 episodios MADDPG ya supera al RBM en más de 100×.

8. Cómo se usa en la simulación

Los benchmarks son llamados desde case_study_1.py cuando el parámetro algo es "rbm" o "madqn":

# En case_study_1.py:
from .benchmarks import RBMAgent, MADQNSystem

if algo == "rbm":
    system = RBMAgent()
    # En cada paso: actions = system.select_actions(demand, pv, wt, charges)

elif algo == "madqn":
    system = MADQNSystem(obs_dim=obs_space.shape[0], n_agents=3)
    # En cada paso: actions = system.select_actions(obs_list)
    # Tras cada paso: system.push(...) ; system.update()

Desde la CLI o el notebook:

# Terminal
python run_simulation.py --algo rbm   --case 1 --episodes 200
python run_simulation.py --algo madqn --case 1 --episodes 200

# Comparación completa (Tabla 3 del paper)
python run_simulation.py --compare --case 1 --episodes 200
⚠️
Case Study 2 no usa benchmarks: El RBM y MADQN no están implementados para CS2 (HESS + MGA + 5 xMGs), ya que el paper tampoco los evalúa en ese escenario. Solo DDPG/D3PG/TD3/MADDPG/MAD3PG/MATD3 soportan el caso de 9 agentes con la subasta de energía.
