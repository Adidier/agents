    1. Stack PyTorch del proyecto

    Todo el aprendizaje por refuerzo usa PyTorch puro. No se usa ningún framework de RL de alto nivel (no Stable-Baselines, no RLlib). Esto permite replicar exactamente la arquitectura del paper.

    torch.nn.Module

    Base de todas las redes neuronales. Cada red hereda de nn.Module y define su grafo computacional en forward().

    torch.optim.Adam

    Optimizador para todos los parámetros (actor y crítico). Learning rates exactos del paper: actor 5e-4, crítico 1e-3.

    torch.nn.functional

    Se usa F.mse_loss, F.log_softmax, F.relu. Operaciones de forward puras sin estado.

    nn.utils.clip_grad_norm_

    Gradient clipping a norma 1.0 en todos los updates. Previene explosión de gradientes en episodios largos (168 steps).

    2. MLP base — 2 capas ocultas × 256 neuronas

    Tanto el actor como el crítico comparten la misma topología base: un Multi-Layer Perceptron con dos capas ocultas de 256 neuronas y activación ReLU. Este número viene directamente de la Tabla 2 del paper ("Hidden layers: 256 units"). 

    # networks.py — función helper compartida por Actor y Critic
def _mlp_body(input_dim: int, hidden_dim: int) -> nn.Sequential:
    """Dos capas ocultas ReLU de 256 unidades (Table 2 del paper)."""
    return nn.Sequential(
        nn.Linear(input_dim, hidden_dim), nn.ReLU(),   # capa 1: obs → 256
        nn.Linear(hidden_dim, hidden_dim), nn.ReLU(),  # capa 2: 256 → 256
    )

    3. Red Actor μ_φ(s) → acción continua ∈ (−1, 1)

    El actor implementa la política determinista μ_φ(s). Dado el estado de observación local, produce una acción continua en (−1, 1) que representa la fracción de carga/descarga del ESS (o la acción de compra/venta para MGA y xMGs). 

    class Actor(nn.Module):
    def __init__(self, obs_dim, action_dim, hidden_dim=256):
        self.body = _mlp_body(obs_dim, hidden_dim)      # MLP base (2×256 ReLU)
        self.head = NoisyLinear(hidden_dim, action_dim) # exploración parametrizada

    def forward(self, obs):
        return torch.tanh(self.head(self.body(obs)))   # salida acotada (−1, 1)

    4. Capa NoisyLinear — exploración parametrizada

    En lugar de usar una política ε-greedy o añadir ruido Ornstein-Uhlenbeck, el paper elige NoisyNet (Fortunato et al. 2017): el ruido de exploración está aprendido dentro de los propios pesos de la red. Así la exploración se adapta automáticamente durante el entrenamiento.

    Cómo funciona matemáticamente

    Cada peso de la capa tiene dos componentes: un valor medio μ y una desviación σ. Durante training se muestrea ruido factorizado y el peso efectivo es:

    w = μ_w + σ_w ⊙ ε_w , b = μ_b + σ_b ⊙ ε_b
    ε ~ función de ruido factorizado (Fortunato et al. 2017, Ec. factorised noise)
    class NoisyLinear(nn.Module):
    def __init__(self, in_features, out_features, sigma_init=0.5):
        # Parámetros aprendibles: μ y σ para pesos y bias
        self.weight_mu    = nn.Parameter(torch.empty(out_features, in_features))
        self.weight_sigma = nn.Parameter(torch.empty(out_features, in_features))
        self.bias_mu      = nn.Parameter(torch.empty(out_features))
        self.bias_sigma   = nn.Parameter(torch.empty(out_features))

        # Buffers de ruido (re-muestreados en cada forward)
        self.register_buffer("weight_eps", torch.empty(out_features, in_features))
        self.register_buffer("bias_eps",   torch.empty(out_features))

    def forward(self, x):
        if self.training:
            w = self.weight_mu + self.weight_sigma * self.weight_eps  # ruido activo
            b = self.bias_mu   + self.bias_sigma   * self.bias_eps
        else:
            w = self.weight_mu   # inferencia: solo la media (determinista)
            b = self.bias_mu
        return F.linear(x, w, b)

    Ruido factorizado (eficiencia computacional)

    @staticmethod
def _scale_noise(size):
    x = torch.randn(size)
    return x.sign() * x.abs().sqrt()   # f(x) = sgn(x)·√|x|

def sample_noise(self):
    eps_in  = self._scale_noise(self.in_features)   # vector p
    eps_out = self._scale_noise(self.out_features)  # vector q
    self.weight_eps.copy_(eps_out.outer(eps_in))    # outer product p×q → matriz
    self.bias_eps.copy_(eps_out)
    📄
    Paper §3.2 — Exploration: "We employ NoisyNet to replace the standard ε-greedy or Ornstein-Uhlenbeck noise. The noise parameters σ are learned alongside the policy parameters φ by gradient descent." La implementación es exactamente Fortunato et al. factorised noise.
    ℹ️
    Durante evaluación (model.eval()) el código usa solo la media weight_mu, sin ruido. Esto corresponde al comportamiento greedy determinista que el paper describe para los episodios de evaluación.

    5. Red Crítico Q_θ(s, a) — crítico estándar DDPG/TD3

    El crítico estima el valor de largo plazo de tomar la acción a en el estado s. Es un MLP que concatena estado y acción como entrada y produce un escalar Q. 

    class Critic(nn.Module):
    def __init__(self, obs_dim, action_dim, hidden_dim=256):
        self.net = nn.Sequential(
            nn.Linear(obs_dim + action_dim, hidden_dim), nn.ReLU(), # entrada: [s‖a]
            nn.Linear(hidden_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, 1),                                # salida: Q escalar
        )

    def forward(self, obs, action):
        return self.net(torch.cat([obs, action], dim=-1))

    ¿Por qué concatenar estado y acción al inicio?

    Una alternativa clásica es inyectar la acción en la segunda capa. El paper sigue la variante donde [obs ‖ action] se concatenan antes de la primera capa, permitiendo que la red aprenda interacciones estado-acción desde el principio. Esta es la arquitectura descrita en Lillicrap et al. (DDPG original).

    TD3: dos críticos gemelos

    Para TD3, se instancian dos críticos independientes (Critic1 y Critic2) con la misma arquitectura. En el cálculo del target se usa el mínimo de ambos para reducir la sobreestimación:

    y = r + γ · min(Q₁_target(s', ã), Q₂_target(s', ã))
    TD3 twin critics — Fujimoto et al. 2018

    6. MADDPGCritic — Crítico centralizado multi-agente

    Esta es la pieza clave de MADDPG y la que lo diferencia del simple DDPG. El crítico de cada agente recibe las observaciones y acciones de todos los agentes durante el entrenamiento. 

    class MADDPGCritic(nn.Module):
    def __init__(self, total_obs_dim, total_action_dim, hidden_dim=256):
        # total_obs_dim    = sum de obs de TODOS los agentes
        # total_action_dim = sum de acciones de TODOS los agentes
        self.net = nn.Sequential(
            nn.Linear(total_obs_dim + total_action_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, 1),
        )

    def forward(self, obs_all, actions_all):
        return self.net(torch.cat([obs_all, actions_all], dim=-1))

    Cada agente tiene SU PROPIO crítico centralizado

    No hay un único crítico compartido. Hay N críticos (uno por agente), cada uno entrenado para estimar la recompensa de su agente dado el estado y acciones globales. Esto se instancia en maddpg.py: 

    # Dentro de _MADDPGAgent.__init__():

self.critic  = MADDPGCritic(total_obs_dim, total_action_dim, cfg.hidden_dim)  # visión global

    7. DistributionalCritic — D3PG (C51-style)

    Para el algoritmo D3PG (Distributed Distributional DDPG), el crítico no produce un escalar Q sino una distribución de probabilidad sobre valores de retorno posibles. Esto es el enfoque C51 de Bellemare et al. 

    class DistributionalCritic(nn.Module):
    def __init__(self, obs_dim, action_dim, n_atoms=51, hidden_dim=256):
        self.n_atoms = n_atoms
        self.net = nn.Sequential(
            nn.Linear(obs_dim + action_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, n_atoms),   # salida: n_atoms logits
        )

    def forward(self, obs, action):
        logits = self.net(torch.cat([obs, action], dim=-1))
        return F.log_softmax(logits, dim=-1)   # distribución log-prob

    def q_value(self, obs, action, support):
        probs = self.probs(obs, action)         # (batch, 51)
        return (probs * support.unsqueeze(0)).sum(dim=-1)  # E[Z] = Σ p_i·z_i

    8. _DQNNetwork — Benchmark MADQN

    El benchmark MADQN usa la misma topología MLP pero con una diferencia crucial: la salida no es una acción sino un valor Q por cada acción discreta. Con n_actions=5 la red devuelve 5 Q-values. 

    class _DQNNetwork(nn.Module):
    def __init__(self, obs_dim, n_actions, hidden_dim=256):
        self.net = nn.Sequential(
            nn.Linear(obs_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim), nn.ReLU(),
            nn.Linear(hidden_dim, n_actions),   # Q(s,0)…Q(s,4)
        )                                       # sin tanh: valores sin acotar

    9. CTDE — Centralised Training, Decentralised Execution

    Este es el paradigma fundamental del paper. Las redes PyTorch reflejan esta dualidad directamente en su diseño:

    🎓 Entrenamiento centralizado

    • El crítico recibe obs_all + actions_all
    • Acceso a información global del entorno
    • Loss calculada con Q-values globales
    • Gradientes del actor dependen del crítico global
    • Solo ocurre offline / en replay buffer

    ⚡ Ejecución descentralizada

    • El actor recibe solo obs_i local
    • No conoce acciones de otros agentes
    • Infiere a tiempo real con solo sensores propios
    • Realista: cada ESS o microrred opera de forma autónoma
    • En evaluación: actor.eval() → sin ruido
    # Durante ENTRENAMIENTO (update() en maddpg.py):
# obs_all = cat([obs_0, obs_1, ..., obs_N])  ← visión global
q_current = agent.critic(obs_all, actions_all)        # crítico: info global

# Durante EJECUCIÓN (select_actions() en maddpg.py):
action_i = agent.actor(obs_i)     # actor: solo observación propia
    📄
    Paper §3.3: "During execution, each agent uses only its local observations. The centralised critic is used only during training and discarded at deployment." Esta separación está implementada literalmente: el actor solo necesita obs_i en producción.

    10. Ciclo de entrenamiento PyTorch — paso a paso

    Este es el flujo completo de un paso de actualización en MADDPG:

    1
    Muestrear mini-batch del replay buffer
    batch = replay_buffer.sample(batch_size=64)
    Contiene (obs_all, actions_all, rewards, next_obs_all, dones).
    2
    Calcular target Q con redes target (frozen)
    next_actions_i = actor_target_i(next_obs_i) → para cada agente
    q_target = r_i + γ · critic_target_i(next_obs_all, next_actions_all)
    3
    Loss del crítico (MSE) + backprop
    q_current = critic_i(obs_all, actions_all)
    critic_loss = F.mse_loss(q_current, q_target)
    critic_opt.zero_grad(); critic_loss.backward(); clip_grad_norm_; critic_opt.step()
    4
    Loss del actor (policy gradient) + backprop
    actions_i_new = actor_i(obs_i) ← acciones actuales del agente i
    actor_loss = −critic_i(obs_all, actions_all_replaced_i).mean()
    Maximizar Q → minimizar −Q.
    5
    Soft update de target networks (Polyak)
    θ_target ← τ · θ_online + (1−τ) · θ_target con τ=0.005
    Actualización suave para estabilidad.
    6
    Re-muestrear ruido NoisyNet
    actor.head.sample_noise() → nuevas ε para la próxima acción.
    Garantiza exploración variada en cada paso.
